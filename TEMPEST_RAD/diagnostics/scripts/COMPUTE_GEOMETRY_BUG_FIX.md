# CRITICAL BUG: compute_geometry() sun_phases Interpretation

## Problem

The `compute_geometry()` function in `simulator.py` returns `sun_phases` as **local hour angles** for each facet (0° = local noon, 180° = local midnight). But the LUT expects a **global body rotation phase** that's the same for all facets.

## Evidence

In the notebook, when querying the LUT at "zero phase angle" (observer = sun):
```
Visible facets: 640/1280
Sun phases (visible): min=0.0°, max=180.0°, std=52.1°
```

Every facet gets a different `sun_phase` value! But all facets on the body should share the same rotation phase at any given instant.

## Why This Causes Wrong Results

The LUT was generated by simulating a single crater through one full rotation cycle (0-360°). At each timestep, the entire crater experienced the same rotation phase. The LUT dimensions are:

```
(Theta, Angle, Latitude, TIME, Wave, Emission, Azimuth)
```

The `TIME` axis represents the body's rotation phase (0-360°), NOT per-facet local times.

When we pass different `sun_phases` for each facet:
- Facet at local noon: gets `sun_phase=0°` → looks up t_idx=0 in LUT
- Facet at local midnight: gets `sun_phase=180°` → looks up t_idx=45 in LUT  
- But both facets exist **at the same instant** on the rotating body!

This causes random/incorrect LUT lookups, resulting in factors < 1.0 everywhere.

## Solution

The `TIME` parameter in the LUT already accounts for thermal history through the rotation cycle. The **latitude** parameter captures how different parts of the body (equator vs poles) experience different thermal environments.

**Option 1: Use a single global time for all facets**

Calculate the body's rotation phase from the simulation time:

```python
def compute_geometry_fixed(facets, sun_vec, obs_vec, rot_axis, simulation_time_hours, period_hours):
    """
    Added parameters:
    - simulation_time_hours: Current time in the simulation
    - period_hours: Body rotation period
    
    Returns sun_phase as a SCALAR for all facets
    """
    # Calculate latitudes, emissions, azimuths as before
    # ...
    
    # Global body rotation phase (same for all facets)
    # Assuming time=0 corresponds to a reference longitude pointing at the sun
    body_rotation_phase = (simulation_time_hours / period_hours * 360.0) % 360.0
    
    # All facets get the same phase
    sun_phases = np.full(n_facets, body_rotation_phase)
    
    return latitudes, sun_phases, emissions, azimuths
```

**Option 2: Rethink LUT generation**

If we really need per-facet local times, the LUT generation needs to be completely restructured. Each (latitude, local_time) combination would need separate thermal simulations, drastically increasing LUT size.

## Recommended Fix

**Option 1** is correct. The LUT already has sufficient dimensions to capture thermal variation:
- **Latitude**: Different thermal environments (pole vs equator)  
- **TIME (body phase)**: Thermal history through rotation
- **Emission/Azimuth**: Viewing geometry effects

The local hour angle of each facet is implicitly captured by the combination of:
- Its **latitude** (where it sits on the body)
- The **body rotation phase** (where the body is in its cycle)
- The **emission angle** (whether we're seeing it at a steep or grazing angle)

## Implementation

1. Modify `simulator.py::compute_geometry()`:
   - Add `simulation_time_hours` and `period_hours` parameters
   - Return a single scalar `body_rotation_phase` instead of per-facet values
   - Or broadcast the scalar to an array for compatibility

2. Update all callers in `retrieval_analysis.ipynb`:
   - Pass the current simulation time and period to `compute_geometry()`
   - Use `time_hours` parameter that's already available

3. Alternative quick fix for the notebook:
   - After calling `compute_geometry()`, override sun_phases:
   ```python
   # Quick fix: Use simulation time instead of local hour angles
   body_phase = (time_hours / period_hours * 360.0) % 360.0
   phases[:] = body_phase  # Same value for all facets
   ```
